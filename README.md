# Лабораторная работа №2 — Резервное копирование и восстановление

**Вариант:** 70219  
**Узлы:** pg116 (основной) → pg117 (резервный)  
**СУБД:** PostgreSQL 16.4, FreeBSD 14.1

## Что сделано

### Этап 1. Резервное копирование

Настроен холодный rsync-бэкап с pg116 на pg117:

- СУБД останавливается, затем `rsync -az` копирует PGDATA и оба табличных пространства
- На pg117 хранятся 14 последних копий, пятнадцатая вытесняет самую старую
- Запускается по cron ежедневно в 03:00

```
scripts/task1/backup.sh   — сам скрипт бэкапа
```

Расчёт объёма за 30 дней при росте 700 МБ/сут и хранении 14 копий: ~225 ГБ.

### Этап 2. Потеря основного узла

Поднята СУБД на pg117 из последней rsync-копии:

- Скопированы PGDATA и табличные пространства
- Исправлены симлинки `pg_tblspc/<OID>` под пути pg117
- Убран `unix_socket_directories` pg116 из `postgresql.auto.conf`
- Добавлены локальные настройки pg117 в `postgresql.conf`

```
scripts/task2/restore-on-backup.sh
```

### Этап 3. Повреждение конфигурации

Симулирована потеря `postgresql.conf`, `postgresql.auto.conf`, `pg_hba.conf`:

- СУБД не запускается без конфигурации
- Конфиги восстановлены из rsync-бэкапа через `rsync` с pg117
- Табличные пространства перенесены в `~/ts_restore/` (исходное расположение «недоступно»), симлинки обновлены

```
scripts/task3/01-simulate.sh   — удаление конфигов
scripts/task3/02-restore.sh    — восстановление
```

### Этап 4. PITR через WAL-архив

Восстановление данных на момент времени до `DROP TABLE`:

- Включены `wal_level=replica`, `archive_mode=on`, взят `pg_basebackup`
- В таблицы вставлены тестовые строки, зафиксировано время
- Выполнены `pg_switch_wal()` и `DROP TABLE data_ydp10, data_zcc31`
- Из базового снимка + WAL-архива восстановлено состояние на метку времени до удаления

```
scripts/task4/01-config.sh     — WAL archiving + pg_basebackup
scripts/task4/02-insert.sh     — вставка строк
scripts/task4/03-error.sh      — DROP TABLE
scripts/task4/04-recovery.sh   — PITR-восстановление
scripts/task4/05-end-recovery.sh — выход из режима recovery
```

## Вопросы к защите

### Резервные копии: назначение, виды

**Назначение** — защита от потери данных при сбоях оборудования, ошибках ПО, случайных или злонамеренных действиях. Позволяют восстановить состояние БД на определённый момент.

**Локальные vs удалённые:**
- *Локальные* — быстрее создаются и восстанавливаются, но не защищают от физической потери носителя (пожар, кража, выход из строя стойки)
- *Удалённые* — требуют сетевого канала, медленнее, зато изолированы от аварий на основной площадке; в данной работе pg117 — удалённый узел

**Полные vs инкрементные:**
- *Полные* — каждый раз копируется весь объём данных; просты в восстановлении, дороги по месту
- *Инкрементные* — копируются только изменения с момента предыдущей копии; экономят место, но восстановление сложнее (нужна цепочка)
- *Дифференциальные* — изменения от последней *полной* копии; компромисс между ними

В работе используются полные копии (rsync каждый раз переписывает всё).

**Холодные vs горячие:**
- *Холодные* — СУБД остановлена на время копирования; гарантированная консистентность, простота реализации, но есть простой
- *Горячие* — копирование на работающей СУБД; нет простоя, но требует специальных механизмов (WAL, snapshot) для обеспечения консистентности

В работе — холодный бэкап (этап 1). PITR (этап 4) — разновидность горячего подхода.

### Сравнение подходов: SQL Dump, FS Level Backup, Continuous Archiving

| | SQL Dump (`pg_dump`) | FS Backup (`rsync`) | Continuous Archiving (WAL) |
|---|---|---|---|
| **Как работает** | Логический дамп: читает данные через SQL | Побайтовая копия файлов PGDATA | Базовый снимок + непрерывный поток WAL |
| **Горячий/холодный** | Горячий | Холодный (в данной работе) | Горячий |
| **Консистентность** | Транзакционно-консистентный снимок | Только при остановленной СУБД | Полная: любой момент времени |
| **PITR** | Нет | Нет | Да — до секунды |
| **Скорость создания** | Медленно (чтение через SQL) | Быстро (копирование файлов) | Непрерывно, нет пика нагрузки |
| **Скорость восстановления** | Медленно (воспроизведение INSERT) | Быстро (копирование файлов) | Зависит от объёма WAL для воспроизведения |
| **Объём хранения** | Меньше (нет bloat, можно сжать) | Больше (весь кластер, bloat включён) | Средний (снимок + дельта WAL) |
| **Кросс-версионность** | Да (можно мигрировать между версиями) | Нет (только та же мажорная версия) | Нет |
| **Применимость** | Миграция, разовые дампы, небольшие БД | Простой DR без требований к RPO | Продуктив с требованием минимального RPO |

### Как рассчитать объём хранилища для резервных копий?

Для полных копий с ротацией N штук:

```
V = Σ(d=D-N+1 .. D) S(d)
```

где `S(d) = S₀ + d × δ` — размер копии в день d, δ — суточный прирост данных.

**От чего зависит:**
- **Размер БД** — больше данных, больше каждая копия
- **Прирост данных** — скорость роста S(d)
- **autovacuum** — при отключённом autovacuum мёртвые кортежи накапливаются в файлах и попадают в каждую копию
- **Число хранимых копий** — линейно влияет на итоговый объём
- **Сжатие** — `rsync -z` или `pg_dump -Fc` уменьшают объём в 2–5 раз для типичных данных
- **Дедупликация** — блочная дедупликация (ZFS, Btrfs) эффективна для инкрементальных изменений
- **Табличные пространства** — нужно учитывать все директории, не только PGDATA

В данной работе: S₀ ≈ 30 МБ, δ = 700 МБ/сут, N = 14 → **~225 ГБ** на 30-й день.

### Как рассчитать требуемую пропускную способность канала?

```
B = S(d) / T_window
```

где `T_window` — допустимое время окна бэкапа (обычно ночное).

**Пример:** копия 11 ГБ, окно 4 часа → нужно ~7.7 Мбит/с (≈ 1 МБ/с). При росте до 21 ГБ на 30-й день — уже ~15 Мбит/с.

**От чего зависит:**
- **Размер копии** — растёт со временем
- **Длительность окна** — ночной бэкап ограничен окном простоя (холодный) или технологическим окном
- **Сжатие (`rsync -z`)** — снижает трафик, но нагружает CPU
- **Latency канала** — rsync чувствителен к round-trip time при работе с большим числом мелких файлов
- **Параллелизм** — можно rsync-ить PGDATA и табличные пространства параллельно
- **Overhead SSH** — шифрование добавляет ~10–15% нагрузки на CPU, незначительно снижает throughput

### Какие факторы влияют на скорость восстановления?

**1. Метод резервного копирования:**
- FS Backup (rsync) — восстановление почти линейно зависит от объёма, нет дополнительных шагов
- SQL Dump — нужно воспроизвести все INSERT, строить индексы; в разы медленнее
- WAL PITR — к времени rsync добавляется воспроизведение WAL-архива (зависит от объёма изменений за период)

**2. Размер данных** — больше данных, дольше копировать файлы по сети

**3. Сетевой канал** — при удалённом восстановлении узкое место чаще всего сеть

**4. Дисковая подсистема** — последовательная запись (восстановление) быстрее случайной; SSD vs HDD существенно

**5. Количество файлов** — много мелких файлов в pg_tblspc замедляют rsync из-за syscall overhead

**6. Необходимость воспроизведения WAL** — при PITR нужно «прокрутить» все WAL-сегменты от снимка до целевой метки; чем длиннее период, тем дольше

**7. Инициализация индексов** — при восстановлении из SQL Dump индексы строятся заново; при FS-копии они уже на месте

**8. Сложность процедуры** — ручные шаги (пересоздание симлинков, правка конфигов) добавляют время и риск ошибки

### Дополнительные вопросы, которые могут задать

**Как уменьшить объём резервных копий?**
- Включить `autovacuum` — мёртвые кортежи освобождаются, в копию попадает меньше мусора
- Использовать сжатие: `rsync -z`, `pg_dump -Fc` или архивацию tar.gz
- Перейти с полных копий на инкрементные (например, `pgbackrest` или WAL-архивирование)
- Блочная дедупликация на уровне ФС резервного узла (ZFS `dedup`)
- Уменьшить число хранимых копий или сократить `fill_factor` таблиц

**Почему rsync с `--delete`, а не просто копирование?**  
Без `--delete` старые удалённые из исходника файлы накапливались бы в бэкапе и постепенно занимали место. `--delete` синхронизирует состояние один-в-один.

**Зачем `pg_switch_wal()` перед DROP TABLE?**  
WAL-сегмент архивируется только когда он заполнен или переключён. Без явного переключения последние изменения (INSERT) могли остаться в текущем незакрытом сегменте и не попасть в архив к моменту восстановления.

**Почему симлинки pg_tblspc нужно пересоздавать после FS-бэкапа?**  
`pg_basebackup -Ft` сохраняет табличные пространства как отдельные tarballs, а в PGDATA/pg_tblspc остаётся пустая директория. При rsync-бэкапе симлинки копируются, но указывают на абсолютные пути исходного хоста — на другом узле эти пути не существуют.

**Что будет, если восстановить FS-бэкап без остановки СУБД?**  
Получится несогласованная копия: часть страниц будет записана в середине транзакции, WAL на момент копирования мог не быть синхронизирован с данными. СУБД обнаружит это при старте и потребует наличия WAL для восстановления (или вовсе откажется стартовать).

**Чем `recovery_target_action = promote` отличается от `pause`?**  
`pause` останавливает воспроизведение WAL и ждёт команды `pg_wal_replay_resume()` — удобно для проверки данных на нужный момент. `promote` сразу переводит кластер в режим primary, создаёт новую timeline и принимает запись.
